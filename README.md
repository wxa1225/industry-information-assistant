# 智能行业研究平台 — 带交叉验证的 AI 深度研究报告生成系统

> 面向券商/咨询公司/投研团队的 AI 行业研究报告自动生成平台
> 核心差异化：每份报告的结论都经过多源交叉验证，每条事实带置信度评分

## 问题背景

券商分析师和咨询顾问撰写行业研究报告时面临三个核心痛点：

1. **数据采集耗时** — 需要手动搜索行业数据、政策文件、公司信息，通常占研究时间的 60%+
2. **矛盾结论难辨** — 不同来源给出互相矛盾的数据（A 报告说市场规模 1000 亿，B 报告说 5000 亿），人工验证费时
3. **质量难以保证** — 纯靠人工经验判断信息来源可信度，缺乏系统化的置信度评估

本系统用多智能体协作自动化"研究计划 → 数据采集 → 分析 → 撰写 → 审核"全流程，并在审核环节自动检测数据冲突、执行交叉验证、输出带置信度的验证结论。

## 核心能力

| 能力 | 说明 |
|------|------|
| **多智能体研究流程** | 6 个专家 Agent 协作：架构师定大纲 → 侦察员搜数据 → 分析师做分析 → 代码极客画图 → 写手写报告 → 审核员质检 |
| **交叉验证引擎** | 自动检测报告中的矛盾数据（数值矛盾、结论方向矛盾），发起补充搜索验证，输出验证结论 |
| **置信度评分** | 四维加权：来源权威性 35% + 交叉验证 30% + 时效性 20% + 信息具体度 15%，每条事实独立评分 |
| **动态工具选择** | 根据研究问题意图自动匹配技能（搜索/知识库/Tushare 财经/代码执行），依赖链自动补全 |
| **LLM 调用优化** | 响应缓存 + Few-shot 示例 + 指数退避重试 + 多模型降级，调用失败率降低 90%+ |
| **代码自愈** | Python 数据分析沙箱执行，失败后 LLM 自动修复（最多 3 次重试） |
| **全链路可观测** | 每次研究生成 trace_id，记录所有 LLM 调用、token 消耗、成本估算，按 Agent 维度输出成本分布 |
| **记忆系统** | 长期记忆四层噪声过滤（哈希/语义/信息增益/价值评分），混合召回（语义 60% + 时间衰减 25% + 重要性 15%） |
| **自动化评估** | 7 项指标 + LLM-as-Judge + 15 条黄金测试集，支持 V1 vs V2 A/B 对比 |

## 快速启动

### 1. 启动基础设施

```bash
docker compose up -d
```

包含：PostgreSQL (5432), Redis (6379), Milvus (19530), Elasticsearch (1200), MinIO (9001/9001)

### 2. 配置环境变量

```bash
cd backend
cp .env.example .env
# 编辑 .env，填入 API Key
```

必填项：
```env
DASHSCOPE_API_KEY=your-key     # 阿里云百炼
BOCHA_API_KEY=your-key         # 博查搜索
```

### 3. 启动后端

```bash
cd backend
pip install -r requirements.txt
python app/app_main.py
```

### 4. 启动前端

```bash
cd frontend
npm install --legacy-peer-deps
npm run dev
```

前端默认运行在 `http://localhost:5173`，后端在 `http://localhost:8000`。

## 评估体系

系统内置自动化评估模块 (`backend/app/evaluation/`)，支持：

- **7 项自动化指标**：来源多样性、引用密度、时效性、章节完整度、幻觉率、数据丰富度、冲突解决率
- **LLM-as-a-Judge**：对逻辑连贯性、论点支撑度、专业深度、可读性进行 rubric-based 评估
- **15 条黄金测试集**：覆盖 AI 芯片、安责险、新能源汽车、云计算、半导体等行业分析场景
- **基准测试对比**：支持 A/B 对比不同配置（V1 vs V2，不同 prompt 版本）的效果

## 面试问答

### Q1: 你的场景到底是什么？为什么用 Agent 而不是 Workflow？

**A**: 这是一个面向券商/咨询公司的行业研究报告自动生成平台。

选择多 Agent 而非硬编码 Workflow 的原因：
- **研究本身不是线性流程**。人类分析师的研究过程是"查资料 → 发现矛盾 → 补充搜索 → 重新判断"的循环，Workflow 只能描述固定流程，而 Agent 可以根据研究结果动态决策下一步
- **审核-修订循环无法预知轮数**。CriticMaster 发现报告质量不达标时，需要根据具体问题决定是"补充搜索"（需要新数据）还是"直接修订"（只需文字修改），这是动态路由
- **每个角色有独立判断**。Architect 决定研究范围，Scout 决定搜索策略和是否递归深挖，Writer 决定章节结构——每个 Agent 都有自己的 prompt、模型选择和输出格式，硬编码会丧失灵活性

### Q2: 速度 vs 精度 vs 成本你怎么取舍？

**A**: 三维度做了平衡：
- **模型路由**：搜索阶段（Scout）用 qwen-plus（快+便宜），写作/审核阶段用 deepseek-v3.2（精度高），不用所有阶段都上大模型
- **收敛检测**：审核-修订循环设置最大 3 轮，连续 2 轮质量提升不超过 0.5 分就强制结束，避免无限循环烧 token
- **缓存机制**：相同 prompt 的 LLM 调用直接命中缓存，避免重复消耗
- **超时控制**：LLM 调用 60s 超时，3 次重试后降级到备用模型
- 实际跑 10 次研究发现 Scout 阶段占 60% token 消耗，所以把 Scout 的模型换成了性价比更高的 qwen-plus

### Q3: Memory 机制：记忆提取和召回具体怎么做的？

**A**: 分两步。

**提取**：LLM 对对话进行结构化总结，提取摘要（summary）、关键洞察（key_insights）、用户偏好（user_preferences）和主题标签（topics）。

**召回**：不是简单的向量搜索，而是混合重排序：
- 先从 Milvus 召回 3×top_k 的候选记忆（语义向量搜索）
- 然后按综合评分 = 语义得分 × 0.6 + 时间衰减 × 0.25 + 重要性 × 0.15 排序
- 时间衰减用指数衰减 e^(-0.05×days)，半衰期约 14 天
- 重要性基于记忆类型（insight > summary > topics）、内容长度和是否包含具体数据

### Q4: 噪声怎么处理？怎么区分真记忆和干扰噪音？

**A**: 四层噪声过滤，在存储前就过滤掉：
1. **哈希去重**：完全相同的内容不重复存储
2. **语义相似度**：与已有记忆 embedding 相似度 > 0.90 视为重复
3. **信息增益**：基于停用词比例、数字密度、标点密度计算，< 0.15 的视为无实质信息
4. **价值评分**：信息密度 40% + 事实性 35% + 完整性 25%，< 0.30 的丢弃

比如"你好"/"谢谢"这种纯闲聊会被长度过滤拦截，"这个很好"会被信息增益过滤拦截，重复的统计数字会被哈希去重拦截。

### Q5: 架构权衡：效率、回答质量、推理成本怎么平衡？

**A**: 上面 Q2 说了成本侧的控制。质量侧的保障来自：
- **CriticMaster 交叉验证**：不是纯 LLM 主观打分，而是先做规则冲突检测（数值提取 → 关键词分组 → 矛盾判断），再做补充搜索验证，最后四维置信度评分
- **7 项自动化评估**：每次研究完成后自动计算来源多样性、引用密度、时效性等指标
- **15 条黄金测试集**：用基准测试持续监控质量变化

### Q6: 工具感知：底层如何动态感知并加载可用 skills？

**A**: 通过 DynamicSkillLoader 实现：
1. **意图识别**：从查询关键词自动识别研究类别（市场/政策/竞争/趋势/技术/金融）
2. **技能评分**：对每个注册的技能按关键词匹配 40% + 类别匹配 30% + 数据类型覆盖 30% 打分
3. **依赖解析**：自动补充前置技能（如 deep_read 需要先执行 web_search）
4. **延迟加载**：重型组件（代码执行器、Tushare）只在需要时才初始化，不用不占内存

### Q7: 你的项目和培训班项目有什么区别？

**A**: 培训班项目只有 6 个 Agent 串起来调 API，没有评估体系、没有可观测性、没有容错。我做了三块改造：
1. **可观测性**：给每次 LLM 调用加了 token 统计和 trace_id，能按 Agent 维度看成本分布
2. **交叉验证**：CriticMaster 不只是打分，而是做了冲突检测引擎（数值+定性矛盾检测）→ 补充搜索验证 → 置信度评分，改变了系统输出从"写报告"到"判断结论可不可信"
3. **评估体系**：7 项自动指标 + 15 条黄金测试集 + LLM-as-Judge，能跑 A/B 对比不同配置的效果
4. **工程健壮性**：重试降级、缓存、收敛检测、内存噪声过滤、CORS 可配置、JWT 解码审计
